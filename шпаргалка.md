### Makefile — все команды

```bash
make help          # список всех команд

# Docker
make up            # запустить все сервисы
make down          # остановить
make build         # пересобрать образы
make restart       # down + build + up
make logs          # логи api и worker (follow)
make logs-all      # логи всех сервисов
make ps            # статус контейнеров

# Разработка
make lint          # go vet для api и worker
make test          # все тесты
make test-api      # тесты только api
make test-worker   # тесты только worker
make build-api     # собрать бинарник api локально
make build-worker  # собрать бинарник worker локально

# Обслуживание
make migrate       # применить DB миграции (рестарт api)
make clean-temp    # удалить /media/temp/*
make gen-secrets   # сгенерировать случайные секреты
```

### Conventional Commits — формат коммитов

```
<тип>(<скоуп>): <описание строчными буквами без точки в конце>
```

**Типы:**

| Тип | Когда |
|---|---|
| `feat` | новая функциональность |
| `fix` | исправление бага |
| `docs` | только документация |
| `refactor` | рефакторинг без изменения поведения |
| `test` | тесты |
| `chore` | зависимости, конфигурация |
| `ci` | CI/CD пайплайн |
| `build` | Dockerfile, Makefile |
| `perf` | производительность |
| `revert` | откат коммита |

**Скоупы:** `api` `worker` `frontend` `docker` `docs` `deps` `auth` `queue` `player` `subtitles` `ffmpeg` `db` `config`

**Примеры правильных коммитов:**
```bash
git commit -m "feat(api): add rate limiting middleware for login endpoint"
git commit -m "fix(worker): handle ffmpeg timeout on corrupted input files"
git commit -m "docs(contracts): update ConvertPayload with attempt field"
git commit -m "chore(deps): bump pgx from v5.7.2 to v5.7.3"
git commit -m "refactor(api): extract job status transitions to separate method"
git commit -m "ci: add github actions workflow for go build and vet"
```

**Примеры неправильных коммитов (hook заблокирует):**
```bash
git commit -m "fix bug"              # нет типа
git commit -m "WIP"                  # нет типа
git commit -m "feat(api): Add thing."  # заглавная + точка
```

**Экстренный обход хука** (использовать осознанно):
```bash
git commit --no-verify -m "wip: временный коммит"
```

Полная документация: `docs/contributing/conventional-commits.md`

## 1) Быстрый локальный старт (API + админка + worker)

### Подготовка `.env`

```bash
cd /Users/robospot/prj/cleaner/converter
cp .env.example .env
```

Минимум, что нужно поменять в `.env`:

```env
POSTGRES_PASSWORD=yourpassword
JWT_SECRET=<вывод openssl rand -hex 32>
PLAYER_API_KEY=any-random-string
ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=yourpassword
QBITTORRENT_PASSWORD=yourpassword
```

Сгенерировать `JWT_SECRET`:

```bash
openssl rand -hex 32
```

`PROWLARR_API_KEY` можно оставить временным до первого входа в UI Prowlarr.

### Сборка и запуск

```bash
# Сборка всех основных сервисов
docker compose build api worker frontend

# Запуск всего стека
docker compose up -d

# Проверка статуса
docker compose ps
```

Ожидаемо через ~60 секунд: `postgres`, `redis`, `prowlarr`, `qbittorrent`, `api`, `worker`, `frontend` в статусе `healthy`.

## 2) Первичная настройка сервисов

### Prowlarr

Открыть UI:

- уже опубликован: `http://localhost:9696`
- если порт не опубликован в вашем compose, временно:

```bash
docker compose run --rm --service-ports -p 9696:9696 prowlarr
```

Дальше:

1. Пройти wizard.
2. Добавить индексеры.
3. Взять `API Key` в `Settings -> General`.
4. Вставить в `.env`:

```env
PROWLARR_API_KEY=<ключ>
```

5. Перезапустить API:

```bash
docker compose restart api
```

### qBittorrent

Открыть UI:

- обычно `http://localhost:8080`
- если порт не опубликован, временно:

```bash
docker compose run --rm --service-ports -p 8080:8080 qbittorrent
```

После входа поменять пароль на значение из `.env` (`QBITTORRENT_PASSWORD`).


## 6) Deploy player на сервер (кратко)

### DNS

```text
A player.ptrack.ink -> 89.167.98.18
```

Проверка:

```bash
dig +short player.ptrack.ink
```

### Сборка и запуск контейнера

```bash
cd /opt/converter/player
cp .env.example .env
```

`/opt/converter/player/.env`:

```env
API_URL=https://api.ptrack.ink
PLAYER_KEY=<тот же ключ, что PLAYER_API_KEY в backend>
NEXT_PUBLIC_VAST_TAG=
```

Запуск:

```bash
cd /opt/converter/player
docker build -t ptrack-player:latest .
docker rm -f ptrack-player 2>/dev/null || true
docker run -d \
  --name ptrack-player \
  --restart unless-stopped \
  -p 127.0.0.1:3100:3000 \
  --env-file .env \
  ptrack-player:latest
```

Проверка:

```bash
curl -I http://127.0.0.1:3100
docker logs --tail=50 ptrack-player
```

### Nginx (идея)

- `player.ptrack.ink` -> reverse proxy на `127.0.0.1:3100`
- затем сертификат:

```bash
sudo certbot --nginx -d player.ptrack.ink
```

## 7) Подключение к БД через DBeaver

### Параметры подключения

| | API сервер (app DB) | Scanner сервер (scanner DB) |
|---|---|---|
| Host | `178.104.100.36` | `213.111.156.183` |
| Port | `5432` | `5432` |
| Database | из `.env` (`POSTGRES_DB`) | `scanner` |
| User | из `.env` (`POSTGRES_USER`) | `scanner` |
| Password | из `.env` (`POSTGRES_PASSWORD`) | `scanner` |

### Доступ разрешён только для IP из UFW

На **API сервере** (178.104.100.36) порт 5432 закрыт для всех кроме явно разрешённых IP.
На **Scanner сервере** (213.111.156.183) UFW неактивен — порт открыт для всех.

### Проверить текущие правила (API сервер)

```bash
ssh -i ~/.ssh/id_rsa_personal root@178.104.100.36 "ufw status | grep 5432"
```

### Добавить новый IP (если сменился)

```bash
NEW_IP="128.0.210.38"   # заменить на свой

# Добавить новое правило
ssh -i ~/.ssh/id_rsa_personal root@178.104.100.36 \
  "ufw allow from $NEW_IP to any port 5432"
```

### Удалить старый IP

```bash
OLD_IP="1.2.3.4"   # IP который нужно убрать

ssh -i ~/.ssh/id_rsa_personal root@178.104.100.36 \
  "ufw delete allow from $OLD_IP to any port 5432"
```

### Посмотреть свой текущий внешний IP

```bash
curl -s ifconfig.me
```

```
                    ┌────────▼────────┐
                    │    FRONTEND     │  Next.js UI (порт 3000)
                    │  178.104.53.215 │

                    ┌────────▼────────┐
                    │   CONVERTER API │  Go HTTP сервер (порт 8000)
                    │  178.104.53.215 │

                         │       │
          ┌──────────────┘       └──────────────────┐
          │ Redis очереди                            │ Scanner API
          │                                          │ (http://scanner:8080)
 ┌────────▼─────────────┐               ┌────────────▼────────────┐

              │                                       │
              │ rclone SFTP                           │ файлы
              │ (копирует файл                        │ в /incoming/
              │  с scanner на converter)              │
              │                                       │
              └──────────────┬────────────────────────┘
                             │
                    ┌────────▼────────┐
                    │  STORAGE SERVER │  45.134.174.84
                    │  /storage/      │  (готовые HLS файлы
                    └─────────────────┘   после конвертации)
```

### Что где живёт

| Компонент | Сервер | Что делает |
|---|---|---|
| **Frontend** | 178.104.53.215 | UI в браузере |
| **Converter API** | 178.104.53.215 | REST API, бизнес-логика |
| **Converter Worker** | 178.104.53.215 | Скачивание, конвертация, перенос |
| **Scanner** | 213.111.156.183 | Мониторинг /incoming/, скачивание публичных URL |
| **Storage** | 45.134.174.84 | SFTP хранилище готовых HLS файлов |
| **Redis** | 178.104.53.215 | Очереди между API и Worker |
| **PostgreSQL** | 178.104.53.215 | БД заданий, фильмов |

### Два пути скачивания фильма

**Путь 1 — Локальный URL (172.x.x.x, 192.168.x.x, 10.x.x.x)**
```
Нажимаешь "Скачать" в UI
→ Converter API → remote_download_queue (Redis)
→ Converter Worker: download_worker скачивает → /media/downloads/
→ convert_queue → convert_worker (FFmpeg → HLS)
→ transfer_worker (rclone → storage 45.134.174.84)
```
Работает потому что converter сервер находится в той же сети что и локальный сервер.

**Путь 2 — Публичный URL (с прокси или без)**
```
Нажимаешь "Скачать" в UI
→ Converter API → Scanner API: POST /api/v1/downloads
→ Scanner: download_worker скачивает → /incoming/ (с прокси если нужно)
→ Scanner: scan_loop замечает файл → регистрирует
→ Converter Worker: ingest_worker claims → rclone copy → /media/downloads/
→ convert_queue → convert_worker → transfer_worker (как выше)
```
