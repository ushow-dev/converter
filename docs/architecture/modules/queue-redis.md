# Модуль: Queue and Redis

## Назначение

Обеспечивает асинхронную коммуникацию API и Worker:

- буферизация задач;
- retry/delay механизмы;
- контроль конкуренции обработки.

## Границы ответственности

Входит в модуль:

- очереди `download_queue` и `convert_queue`;
- visibility timeout и задержки;
- retry counters и dead-letter стратегия.

Не входит в модуль:

- долговременное хранение статусов/истории;
- бизнес-валидация payload;
- хранение пользовательских данных.

## Интерфейсы и зависимости

Входящие:

- publish задач из Core API.

Исходящие:

- consume задач Worker-ом;
- опциональная публикация событий в отдельный канал.

Зависимости:

- Docker network;
- Redis persistence (AOF/RDB) как операционная настройка.

## Контракты данных

Обязательные поля сообщения:

- `job_id`;
- `job_type` (`download`/`convert`);
- `content_type`;
- `attempt`;
- `created_at`;
- `correlation_id`.

## Отказоустойчивость

- At-least-once доставка с идемпотентностью в consumer.
- Retry с экспоненциальным backoff.
- Dead-letter queue после достижения порога ошибок.
- Очистка устаревших сообщений по TTL для технических очередей.

## Наблюдаемость и безопасность

- Метрики depth/lag по каждой очереди.
- Алерты на рост DLQ и backlog.
- Redis не публикуется наружу; доступ только внутри docker network.
- Требуется пароль Redis и отключение небезопасных команд в production-профиле.

## Точки расширения под сериалы

- Очереди уже принимают `content_type`, поэтому добавление `series` не ломает контракт.
- Возможно введение дополнительной очереди orchestration для пакетных задач сериалов, без изменения текущего download/convert контура.
