# Целевая продакшн-архитектура

> Этот документ описывает целевую архитектуру системы для продакшн-развёртывания
> с раздачей контента пользователям в Бангладеш. Текущее состояние серверов —
> временное; данный документ фиксирует к чему нужно прийти.

## Серверы и их роли

| Сервер | Роль | Рекомендуемое железо |
|---|---|---|
| API Server | Go API + Admin UI + PostgreSQL + Redis | 2-4 ядра, 8 GB RAM, 50 GB SSD |
| Source Storage | Scanner + оригинальные видеофайлы + scanner DB | 4 ядра, 8 GB RAM, 10+ TB HDD |
| Converter(s) | ffmpeg workers, CPU-heavy, горизонтально масштабируется | 16-32 ядра, 16 GB RAM, 500 GB SSD |
| Origin Storage | Готовые HLS-файлы, источник для CDN Edge | 4 ядра, 8 GB RAM, 10+ TB HDD |
| CDN Edge | nginx раздача HLS + статика плеера, зона доставки | 8 ядер, 32 GB RAM, 4+ TB NVMe |

## Потоки данных

### Путь 1 — ручное добавление через Admin UI

```
Admin → API создаёт job → Redis convert_queue
     → Converter скачивает + конвертирует
     → записывает HLS в Origin Storage
     → обновляет API DB (storage_key, base_url)
```

### Путь 2 — автоматический ingest через Scanner

```
Файл появляется в /incoming/ на Source Storage
     → scan_loop: GuessIt + TMDB метаданные, качество, дедупликация
     → ingest_worker: rclone copy на Converter
     → Converter конвертирует
     → записывает HLS в Origin Storage
     → обновляет API DB
```

### Путь пользователя

```
① Пользователь открывает плеер
     → CDN Edge отдаёт player JS/HTML (5-30 мс, из Азии)

② Плеер делает один запрос к API (NL)
     → GET /api/player/movie?imdb_id=...
     → API возвращает { hls_url, subtitles, poster }
     → (150-200 мс, один раз при инициализации)

③ Плеер стримит HLS
     → все сегменты идут напрямую с CDN Edge (5-30 мс)
     → API в стриминге не участвует
```

### Cache miss на CDN Edge

```
Пользователь запросил фильм впервые
     → CDN Edge не имеет сегментов в кэше
     → nginx делает proxy_pass на Origin Storage (NL)
     → кэширует локально на SSD/HDD
     → все последующие запросы этого фильма — из кэша
```

## Почему компоненты разделены именно так

**API отдельно от Converter** — FFmpeg при конвертации насыщает все CPU-ядра.
Совместное размещение приводит к задержкам API во время активной конвертации.
Разделение позволяет горизонтально масштабировать воркеры независимо от API.

**Redis на стороне Converter, не API** — воркеры выполняют BLPOP в непрерывном
цикле. Redis должен быть физически близко к воркерам. API обращается к Redis
редко (только при создании задания).

**Pull-caching вместо push** — воркер не отправляет файлы на CDN Edge явно.
CDN Edge сам тянет файлы с Origin Storage при первом запросе и кэширует.
Это снимает нагрузку на исходящий канал конвертера и упрощает логику воркера.

**Плеер на CDN Edge** — player JS/HTML размещается на том же сервере что и
HLS-сегменты. Пользователь получает и плеер, и контент с одного географически
близкого сервера. Единственный запрос в NL — инициализация метаданных, что
незаметно для VOD.

**Source Storage отдельно от Origin Storage** — оригиналы (исходные видеофайлы)
и конвертированные HLS хранятся раздельно. В будущем оригиналы можно
предоставить для скачивания пользователями независимо от HLS-хранилища.

## Узкие места и как с ними работать

| Узкое место | Описание | Решение |
|---|---|---|
| Межконтинентальный transfer | Origin Storage (NL) → CDN Edge (Азия): cache miss медленный | pull-cache: одноразовый transfer, потом из кэша |
| Единственный CDN Edge | SPOF для всех пользователей | второй edge при росте, failback на origin |
| Scanner → Converter rclone | SFTP медленнее локального диска | разместить в одном DC при большом потоке |
| API в NL | +150-200 мс к открытию плеера | некритично для VOD; при масштабе — read-replica в Азии |
| Один Redis | SPOF для очередей | Redis Sentinel или мониторинг с алертами |

## Изменения в коде относительно текущего состояния

Текущий код архитектурно уже готов к этой схеме. Нужны только конфигурационные
изменения, не изменения логики:

| Что делать | Изменения в коде | Изменения в инфре |
|---|---|---|
| Вынести API + DB на отдельный сервер | нет — только env vars | новый сервер, перенос PostgreSQL |
| Вынести Worker на отдельный сервер | нет — только env vars (`DATABASE_URL`, `REDIS_URL`) | новый сервер |
| CDN Edge с pull-cache | убрать rclone push из воркера (упрощение) | nginx `proxy_cache` на Edge |
| Плеер на CDN Edge | нет — `PLAYER_URL` уже параметризован | деплой статики на Edge |
| Второй воркер | нет — те же env vars | новый сервер |

## Целевое географическое размещение

| Сервер | Расположение | Обоснование |
|---|---|---|
| API Server | NL или любой EU/US | управление, не latency-critical |

| Converter(s) | NL | рядом с Source Storage и Origin |
| Origin Storage | NL | источник для CDN Edge |

## Этапы перехода к целевой архитектуре

### Этап 1 — Разделение API и Worker (готово к старту)
- Новый VPS для API + PostgreSQL + Redis
- Converter сервер — только Worker
- Изменения: только `.env` файлы

### Этап 2 — CDN Edge в Азии
- Арендовать сервер в Mumbai или Singapore (Hetzner, OVH)
- Настроить nginx с `proxy_cache`
- Разместить статику плеера
- Изменить `PLAYER_URL` на адрес Edge

### Этап 3 — Горизонтальное масштабирование Converter
- Добавить второй воркер-сервер
- Указать на тот же Redis и PostgreSQL
- Не требует изменений в коде

### Этап 4 — BDIX (при достижении масштаба ~5 000+ одновременных)
- Регистрация юрлица в Бангладеш или партнёр
- Colocation в DC при BDIX (Dhaka)
- Миграция CDN Edge или добавление второго
