# ADR-002: Два отдельных Go-модуля (api + worker)

**Статус:** accepted
**Дата:** 2024-01-01
**Автор:** AI (зафиксировано при аудите 2026-03-16)

---

## Контекст

Система состоит из двух Go-сервисов с разными обязанностями:
- **API** — HTTP сервер, принимает запросы, управляет заданиями
- **Worker** — фоновый процессор, выполняет длительные операции

Нужно решить: один Go-модуль или два отдельных?

## Рассмотренные варианты

### Вариант A: Два отдельных модуля `api/go.mod` и `worker/go.mod` (текущий выбор)
- Плюсы: полная изоляция зависимостей, независимые Docker образы, чёткая граница
- Минусы: дублирование queue payload моделей в обоих модулях

### Вариант B: Один модуль с несколькими `cmd/`
- Плюсы: единый `go.mod`, общие пакеты без дублирования
- Минусы: размытая граница между сервисами, один большой образ

### Вариант C: Go workspace (`go.work`) с shared пакетом
- Плюсы: общие типы без дублирования, раздельные модули
- Минусы: дополнительная сложность, нужна настройка workspace

## Решение

Выбран **Вариант A: два отдельных модуля**.

Изоляция важнее удобства. Каждый сервис имеет свой независимый жизненный цикл, зависимости и образ. Дублирование payload моделей — известный компромисс, управляемый через документацию контрактов.

## Последствия

### Положительные
- API и Worker деплоятся независимо
- Зависимости не пересекаются
- Минимальный Docker образ для каждого

### Отрицательные / компромиссы
- `DownloadPayload`, `ConvertPayload`, `RemoteDownloadPayload` объявлены в обоих модулях
- При изменении payload нужно обновлять оба места — риск расхождения

### Риски
- Расхождение payload моделей при невнимательном рефакторинге

## Ревизия

Пересмотреть в пользу Варианта C (Go workspace) при:
- Появлении третьего Go-сервиса
- Частых breaking changes в payload форматах
- Решении добавить code generation для моделей
