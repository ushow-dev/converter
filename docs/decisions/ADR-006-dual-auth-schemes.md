# ADR-006: Два режима аутентификации (JWT для admin, API Key для player)

**Статус:** accepted
**Дата:** 2024-01-01
**Автор:** AI (зафиксировано при аудите 2026-03-16)

---

## Контекст

API обслуживает два типа клиентов:
1. **Admin UI** — Next.js приложение, управляет заданиями, требует пользовательской аутентификации
2. **Player** — встраиваемый iframe на сторонних сайтах, обращается к `/api/player/*`

Нужна схема аутентификации для каждого.

## Рассмотренные варианты

### Вариант A: Единый JWT для всех клиентов
- Плюсы: единый механизм
- Минусы: плеер не может хранить JWT безопасно (iframe), токен нужно куда-то передавать

### Вариант B: JWT (admin) + статический API Key (player) (текущий выбор)
- Плюсы: простота для player интеграции, API Key конфигурируется раз
- Минусы: API Key — долгоживущий секрет, нет per-user ротации для player

### Вариант C: OAuth2 / OIDC для обоих
- Плюсы: стандарт, refresh tokens, per-user
- Минусы: значительная сложность, избыточно для self-hosted

## Решение

Выбран **Вариант B: JWT для admin, API Key для player**.

- Admin: `Authorization: Bearer <jwt>` (HS256, истекает через N часов)
- Player: `X-Player-Key: <PLAYER_API_KEY>` (статический ключ из env)
- CORS: `*` для player endpoints (embed use case)
- Admin endpoints закрыты от player key

## Последствия

### Положительные
- Простая интеграция player (один ключ, не меняется)
- JWT для admin позволяет добавить roles в будущем
- Чёткое разделение `/api/admin/*` и `/api/player/*`

### Отрицательные / компромиссы
- Утечка `PLAYER_API_KEY` — нужна ротация во всех интеграциях
- Нет per-user аналитики для player (все запросы под одним ключом)
- JWT в localStorage — уязвимость XSS (см. technical-debt.md #2)

### Риски
- При публичном API player — ключ могут извлечь из JS кода iframe

## Ревизия

Пересмотреть при:
- Требовании per-user аналитики для плеера
- Появлении нескольких независимых player клиентов
- Переходе JWT в httpOnly cookies (тогда схема admin усилится)
