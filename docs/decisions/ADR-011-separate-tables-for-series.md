# ADR-011: Separate tables for series support

**Статус:** accepted
**Дата:** 2026-04-01
**Автор:** AI

---

## Контекст

Необходимо добавить поддержку TV-сериалов в систему. Сериалы состоят из сезонов и эпизодов,
каждый эпизод — это отдельный медиа-ассет с собственными HLS-плейлистами, аудиодорожками и субтитрами.

Существующая схема ориентирована исключительно на фильмы: таблица `movies`, связанные `media_jobs`,
`media_assets`, `movie_subtitles`, `media_audio_tracks`.

Ограничения:
- Нельзя ломать существующий код работы с фильмами
- Нужна чёткая модель для навигации по сезонам и эпизодам
- Аудиодорожки должны поддерживаться и для фильмов, и для эпизодов

## Рассмотренные варианты

### Вариант A: Полиморфное расширение таблицы `movies`

Добавить в `movies` поля `is_series`, `season`, `episode`, `series_id` (self-reference).

- Плюсы: минимум новых таблиц, единый слой репозитория
- Минусы: раздутая схема с множеством NULL-полей, сложные запросы для получения иерархии (серия → сезоны → эпизоды), нарушение принципа единственной ответственности таблицы

### Вариант B: Отдельные таблицы — series, seasons, episodes

Создать новые таблицы `series`, `seasons`, `episodes` рядом с существующей `movies`.
Аудиодорожки переиспользовать через дискриминатор `asset_type` в `media_audio_tracks`.

- Плюсы: чистая модель без NULL-sprawl, не ломает существующий код фильмов, ясная семантика иерархии, масштабируется
- Минусы: больше таблиц и эндпоинтов, отдельные репозитории для каждого типа контента

## Решение

Выбран **Вариант B** — отдельные таблицы.

Схема:
- `series` — метаданные сериала (tmdb_id, imdb_id, title, year, poster_url)
- `seasons` — сезоны (series_id FK, season_number)
- `episodes` — эпизоды (season_id FK, episode_number, title, job_id FK, asset_id FK)
- `media_audio_tracks` дополнена полем `asset_type` (`movie` | `episode`) и `asset_id` для полиморфной связи

Миграции: `api/db/migrations/` (добавлены файлы с новыми таблицами).

## Последствия

### Положительные
- Существующий код фильмов (`movies`, `media_jobs`, `media_assets`) не затронут
- Чёткая иерархическая модель упрощает запросы навигации по сериалу
- Аудиодорожки переиспользуются без дублирования логики через `asset_type`
- Независимые репозитории дают ясные зоны ответственности

### Отрицательные / компромиссы
- Больше таблиц и JOIN-ов при построении полного дерева сериала
- Отдельные эндпоинты `/api/admin/series` и `/api/player/series` — дополнительный surface area API
- Worker должен различать jobs для фильмов и эпизодов при конвертации

### Риски
- При большом количестве сериалов JOIN серия→сезоны→эпизоды→ассеты может стать узким местом; при необходимости можно денормализовать или добавить индексы

## Ревизия

Пересмотреть решение при:
- Появлении третьего типа контента (например, документальные серии с иной иерархией)
- Значительном росте нагрузки на запросы каталога (>10k сериалов)
