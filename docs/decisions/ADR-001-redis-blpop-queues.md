# ADR-001: Redis BLPOP для очередей заданий

**Статус:** accepted
**Дата:** 2024-01-01
**Автор:** AI (зафиксировано при аудите 2026-03-16)

---

## Контекст

Система выполняет длительные фоновые задания (загрузка торрентов, конвертация FFmpeg).
Нужен механизм передачи заданий от API к Worker с гарантией, что:
- задание получит ровно один воркер
- при перезапуске воркера задание не потеряется
- возможен параллельный запуск нескольких воркеров

## Рассмотренные варианты

### Вариант A: Redis LIST + BLPOP (текущий выбор)
- Плюсы: простота, нет дополнительных сервисов, гарантия "ровно один потребитель"
- Минусы: нет dead letter queue из коробки, нет ordering гарантий при failover

### Вариант B: Redis Pub/Sub
- Плюсы: broadcast, низкая задержка
- Минусы: нет гарантии доставки, сообщение теряется если воркер офлайн — не подходит

### Вариант C: RabbitMQ или NATS
- Плюсы: зрелые решения, DLQ, retry, ordering
- Минусы: ещё один сервис в инфраструктуре, сложность для текущего масштаба

### Вариант D: PostgreSQL LISTEN/NOTIFY + polling
- Плюсы: нет Redis в зависимостях, ACID
- Минусы: нагрузка на БД, сложнее реализовать backpressure

## Решение

Выбран **Вариант A: Redis BLPOP**.

Redis уже используется в системе, добавление нового сервиса избыточно при текущих объёмах. BLPOP гарантирует атомарное извлечение одним потребителем. Distributed lock через `SET NX` предотвращает дублирование.

Три очереди:
- `download_queue` — загрузка торрентов
- `convert_queue` — FFmpeg конвертация
- `remote_download_queue` — HTTP загрузка

## Последствия

### Положительные
- Нет дополнительных сервисов
- Простая реализация (RPUSH + BLPOP)
- Работает с существующим Redis инстансом

### Отрицательные / компромиссы
- Нет встроенного dead letter queue
- При крэше воркера сообщение в BLPOP теряется (уже извлечено из очереди, но не обработано)
- Нет replay механизма

### Риски
- При > 50 одновременных заданий может потребоваться переход на более зрелый брокер

## Ревизия

Пересмотреть при:
- Нагрузке > 100 активных заданий одновременно
- Необходимости replay или аудита всех сообщений
- Требовании гарантии обработки при крэше воркера
