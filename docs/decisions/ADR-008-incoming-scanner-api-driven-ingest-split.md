# ADR-008: Incoming Scanner — API-Driven Ingest Split

**Статус:** accepted
**Дата:** 2026-03-18
**Автор:** AI

---

## Контекст

Scanner-сервис обнаруживает новые файлы на отдельном хранилище и должен передавать их в конвертер.
Ключевые ограничения:

- Scanner работает на **отдельном сервере** и не имеет прямого доступа к PostgreSQL конвертера.
- Конвертер не имеет прямого сетевого доступа к файловой системе `incoming/` сканера.
- Внешние сервисы не должны писать напрямую в БД или очереди Redis конвертера — только через API.
- Нужна возможность мониторить прогресс копирования, фиксировать ошибки и перезапускать неудачные попытки без прямого доступа к Redis.

## Рассмотренные варианты

### Вариант A: Прямая запись в Redis (push из сканера)
Scanner напрямую делает `RPUSH convert_queue` на сервере конвертера.

- Плюсы: простейшая реализация, нет HTTP-слоя
- Минусы: нарушает service boundary; Redis конвертера становится публично доступным; нет статус-машины для входящих файлов; сложно retry и мониторинг

### Вариант B: Прямая запись в PostgreSQL из сканера
Scanner делает INSERT напрямую в таблицу конвертера.

- Плюсы: надёжнее очереди, транзакционность
- Минусы: тесная связь по БД-схеме; необходимо открыть порт Postgres наружу; нарушает принцип единственного владельца схемы

### Вариант C: API-only статус-машина (выбранный вариант)
Scanner регистрирует файл через `POST /api/ingest/incoming/register`. Ingest worker конвертера атомарно забирает задания через `claim`, копирует файл через rclone, затем вызывает `complete`, который создаёт `media_job` и кладёт в `convert_queue`.

- Плюсы: чёткие service boundaries; единственная точка входа через authenticated API; полная статус-машина с lease TTL и retry; нет прямого доступа к БД/Redis извне
- Минусы: дополнительный HTTP round-trip; ingest worker должен быть на той же машине или иметь rclone remote доступ

## Решение

Выбран **Вариант C** (API-only).

Архитектура разделена на три роли:

1. **Scanner** (внешний сервис) — обнаруживает файл, вызывает `register`.
2. **Ingest Worker** (внутри конвертера) — периодически вызывает `claim`, копирует через rclone, вызывает `complete`.
3. **API** (конвертер) — владеет таблицей `incoming_media_items`, реализует статус-машину, в `complete` создаёт `media_job` и пушит в `convert_queue`.

Доступ к `/api/ingest/*` защищён заголовком `X-Service-Token` (отдельный секрет `INGEST_SERVICE_TOKEN`).

Lease TTL предотвращает зависание заданий при падении ingest worker — `claim` сбрасывает истёкшие leases обратно в `new`.

## Последствия

### Положительные
- Нет прямого внешнего доступа к PostgreSQL или Redis конвертера.
- Полная статус-машина: `new → claiming → copying → copied → completed / failed`.
- Атомарная выдача заданий через `FOR UPDATE SKIP LOCKED` — безопасен для нескольких воркеров.
- Простая наблюдаемость: статусы хранятся в БД, видны через SQL.
- Scanner не зависит от внутренних деталей реализации конвертера.

### Отрицательные / компромиссы
- Три сервиса должны разделять `INGEST_SERVICE_TOKEN`: scanner, API, ingest worker.
- Lease TTL (`INGEST_CLAIM_TTL_SEC`) нужно настроить исходя из ожидаемого времени копирования больших файлов.
- Обратный вызов из конвертера в scanner (перемещение в библиотеку после конвертации) отложен на следующий блок (Block 3) — scanner пока не знает о завершении конвертации.

### Риски
- Если `INGEST_SERVICE_TOKEN` утечёт, внешний сервис может регистрировать произвольные пути.
- При очень больших файлах lease может истечь раньше завершения rclone — TTL нужно мониторить.

## Ревизия

Пересмотреть при:
- Появлении callback-API для уведомления scanner о завершении конвертации (Block 3).
- Необходимости поддержки нескольких scanner-сервисов с разными токенами.
- Переходе на event-driven архитектуру (message broker вместо polling).
