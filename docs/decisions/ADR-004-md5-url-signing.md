# ADR-004: MD5 подписывание медиа-URL

**Статус:** accepted
**Дата:** 2024-01-01
**Автор:** AI (зафиксировано при аудите 2026-03-16)

---

## Контекст

HLS сегменты (`.ts` файлы) отдаются напрямую из файловой системы через nginx или статический сервер.
Нужен механизм, ограничивающий доступ к медиа только авторизованным пользователям,
при этом не проксируя каждый сегмент через API (высокая нагрузка).

## Рассмотренные варианты

### Вариант A: Без защиты (любой знает URL — может смотреть)
- Плюсы: простота
- Минусы: нет контроля доступа

### Вариант B: MD5 secure_link (текущий выбор, опциональный)
- Плюсы: совместим с nginx `ngx_http_secure_link_module`, без проксирования через API, TTL
- Минусы: MD5 криптографически слабый (для доступа, не для целостности — приемлемо), нужен nginx

### Вариант C: JWT подписанные URL (как AWS S3 presigned)
- Плюсы: криптографически сильнее, стандарт
- Минусы: nginx не верифицирует JWT нативно, нужен lua или отдельный auth сервис

### Вариант D: Проксирование через API
- Плюсы: полный контроль, любая схема авторизации
- Минусы: огромная нагрузка на API при стриминге сотен сегментов

## Решение

Выбран **Вариант B: MD5 secure_link**, опциональный (управляется `MEDIA_SIGNING_KEY`).

Формат: `?md5=base64(md5(key + " " + path + " " + expires))&expires=timestamp`

Совместим с nginx `secure_link` "из коробки". Если `MEDIA_SIGNING_KEY` не задан — URL не подписываются (для локальных установок).

## Последствия

### Положительные
- Нет нагрузки на API при стриминге
- Nginx верифицирует нативно — нет дополнительных сервисов
- Легко включить/выключить

### Отрицательные / компромиссы
- MD5 слабее SHA-256, но для access control (не для integrity) — достаточно
- Требует настройки nginx `secure_link` в продакшне
- `MEDIA_SIGNING_KEY` должен быть секретом

### Риски
- Если nginx не настроен — подписи не проверяются (URL всё равно работают)

## Ревизия

Пересмотреть при:
- Требовании фермерской защиты (per-user токены)
- Переходе на CDN с поддержкой JWT (Cloudflare, AWS CloudFront)
