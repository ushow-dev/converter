# ADR-010: P2P HLS streaming через p2p-media-loader

**Статус:** accepted
**Дата:** 2026-03-26
**Автор:** AI

---

## Контекст

CDN/origin bandwidth — основная статья расходов на стриминг. При одновременном просмотре одного контента несколькими зрителями каждый из них скачивает `.ts` сегменты с origin-сервера независимо. P2P позволяет зрителям обмениваться уже скачанными сегментами через WebRTC, снижая нагрузку на origin.

Ограничения:
- Плеер встроен через iframe на сторонних сайтах
- Используется Fluid Player (VAST реклама) + hls.js
- Текущий storage server не использует URL signing (CORS `*`, файлы отдаются напрямую)

## Рассмотренные варианты

### Вариант A: p2p-media-loader (Novage)
- Плюсы: open source (MIT), self-hosted tracker, интеграция с hls.js через mixin, активно поддерживается
- Минусы: нужен собственный tracker-сервер

### Вариант B: CDNBye / hlsjs-p2p-engine
- Плюсы: простая интеграция, managed tracker
- Минусы: коммерческий (free tier с лимитами), vendor lock-in, данные зрителей уходят на внешний сервер

### Вариант C: Peer5
- Плюсы: enterprise-grade, SLA
- Минусы: полностью коммерческий SaaS, высокая стоимость, overkill для текущего масштаба

## Решение

Выбран **Вариант A: p2p-media-loader**, потому что:
1. Полностью open source — нет лимитов на трафик или количество пиров
2. Self-hosted tracker (`wt-tracker`) — данные не уходят третьим сторонам
3. Mixin-подход (`HlsJsP2PEngine.injectMixin(Hls)`) — минимальные изменения в коде плеера
4. Graceful degradation — при недоступности WebRTC плеер работает как раньше (100% HTTP)

## Последствия

### Положительные
- Снижение bandwidth на origin/CDN пропорционально количеству одновременных зрителей
- Нет зависимости от коммерческих сервисов
- Feature flag (`NEXT_PUBLIC_P2P_ENABLED`) — моментальный rollback

### Отрицательные / компромиссы
- Дополнительный сервис (wt-tracker) — +1 контейнер на API-сервере
- P2P эффективен только при 2+ одновременных зрителях одного контента
- WebRTC за строгими NAT/firewall может не работать (fallback на HTTP)

### Риски
- iOS Safari поддерживает WebRTC DataChannels только с iOS 17.2+ — старые устройства получат 100% HTTP
- iframe на сторонних сайтах с `Permissions-Policy` ограничениями может блокировать WebRTC

## Ревизия

Пересмотреть это решение:
- При необходимости STUN/TURN-серверов для улучшения NAT traversal
- При переходе на managed CDN с собственным P2P (Cloudflare, Akamai)
- При масштабировании до тысяч одновременных зрителей (может потребоваться кластер tracker-ов)
