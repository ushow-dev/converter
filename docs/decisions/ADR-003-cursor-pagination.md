# ADR-003: Курсорная пагинация вместо offset

**Статус:** accepted
**Дата:** 2024-01-01
**Автор:** AI (зафиксировано при аудите 2026-03-16)

---

## Контекст

API возвращает списки заданий (`/api/admin/jobs`) и фильмов (`/api/admin/movies`).
Нужна стратегия пагинации для эффективной работы при росте данных.

## Рассмотренные варианты

### Вариант A: Offset пагинация (`LIMIT N OFFSET M`)
- Плюсы: просто реализовать, привычно для клиентов
- Минусы: при вставке новой записи страница "сдвигается", дубликаты/пропуски при реальном времени; `OFFSET 10000` читает 10000 строк

### Вариант B: Курсорная пагинация по `job_id` (текущий выбор)
- Плюсы: стабильные страницы, O(log N) через индекс, нет дубликатов при append
- Минусы: нельзя перейти на произвольную страницу, нужен индекс по cursor полю

### Вариант C: Keyset пагинация по `(created_at, id)`
- Плюсы: стабильная сортировка по времени
- Минусы: сложнее реализовать составной cursor

## Решение

Выбран **Вариант B: курсорная пагинация по `job_id`**.

Задания и фильмы добавляются append-only. Клиент передаёт `cursor=last_job_id`, API возвращает записи с `job_id < cursor`. Индекс по `job_id` обеспечивает O(log N).

## Последствия

### Положительные
- Производительность не деградирует с ростом таблицы
- Нет дубликатов при одновременных вставках

### Отрицательные / компромиссы
- Нет "прыжка на страницу N"
- Клиент обязан хранить cursor между запросами

### Риски
- Если потребуется сортировка по полям кроме `job_id` — нужна ревизия

## Ревизия

Пересмотреть при:
- Требовании произвольного доступа к страницам (UI с номерами страниц)
- Сортировке по `created_at` или другим полям
