# ADR-007: Удалённое хранилище медиафайлов через rclone

**Статус:** accepted
**Дата:** 2026-03-17
**Автор:** AI

---

## Контекст

Сконвертированные HLS-файлы накапливаются на сервере обработки (где запущен worker). Дисковое пространство ограничено — нужно переносить готовые фильмы на более ёмкий удалённый сервер.

Особенности ситуации:
- Удалённый сервер изначально не имеет веб-сервера; прокси будет настроен позже
- До запуска прокси `base_url` для удалённого хранилища будет пустым — плеер должен продолжать работать, используя локальный путь
- Нужна читаемая структура каталогов на удалённом хранилище (для ручного управления)
- В будущем возможна поддержка S3-совместимых хранилищ

## Рассмотренные варианты

### Вариант A: rsync через SSH
- Плюсы: стандартный инструмент, нет зависимостей
- Минусы: нет встроенной поддержки S3/облаков; требует SSH-доступа; нет `--checksum` без дополнительных настроек

### Вариант B: rclone
- Плюсы: единый инструмент для SSH/SFTP, S3, Backblaze, GCS и других бэкендов; `rclone move` атомарно переносит файлы; широко известен в self-hosted сообществе
- Минусы: дополнительная зависимость в Dockerfile worker

### Вариант C: Прямая загрузка через HTTP/S3 SDK
- Плюсы: нет внешних процессов
- Минусы: нужно реализовывать для каждого провайдера отдельно; значительный объём кода

## Решение

Выбран **Вариант B: rclone**.

- После успешной HLS-конвертации converter кладёт сообщение в `transfer_queue`
- `TransferWorker` (consumer) запускает `rclone move` для переноса файлов на удалённый сервер
- Бэкенды хранилищ описаны в таблице `storage_locations`; поле `base_url` может быть пустым до готовности прокси
- Плеер (`api/internal/handler/player.go`) разрешает `base_url` по `storage_location_id` фильма; если `base_url` пустой — использует локальный `MEDIA_BASE_URL`
- Папки фильмов на удалённом хранилище именуются `Title (Year)` для читаемости при ручном управлении
- Если переменная `RCLONE_REMOTE` не задана, фильмы остаются локально

## Последствия

### Положительные
- Масштабируется на любой бэкенд через rclone-конфиг без изменения кода
- Человекочитаемые пути (`Title (Year)`) упрощают ручное управление
- Смена домена = один SQL UPDATE в `storage_locations.base_url`
- Поддержка S3 требует только добавления новой строки в `storage_locations` и новой конфигурации rclone remote

### Отрицательные / компромиссы
- При незаданном `RCLONE_REMOTE` файлы остаются на локальном диске worker — место не освобождается
- rclone установлен в Dockerfile worker, незначительно увеличивает размер образа
- Перенос выполняется асинхронно — между завершением конвертации и завершением переноса фильм доступен только локально

### Риски
- При сбое `rclone move` файлы остаются локально; нужен мониторинг очереди `transfer_queue`
- Если proxy на удалённом сервере ещё не готов, а `base_url` уже заполнен — стриминг сломается

## Ревизия

Пересмотреть при:
- Добавлении нескольких одновременных удалённых хранилищ (сейчас предполагается одно активное)
- Переходе на S3 как основное хранилище (возможно, прямая загрузка через SDK окажется эффективнее)
- Появлении требований к шифрованию данных при передаче (rclone поддерживает crypt remote)
